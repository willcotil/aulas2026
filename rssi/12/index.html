TCP, UDP e a Engenharia das Portas Lógicas

Onde estamos no Modelo OSI?

L1 a L3: Infraestrutura, Cabos e Roteamento (IP)
L4: Camada de Transporte (O Coração do SO)
L5 a L7: Aplicações e Usuário (HTTP, Game, Mail)

A Camada 4 é a ponte entre a Rede Física e o Software. Sem ela, o computador não saberia para qual programa enviar os dados.

As Missões da Camada de Transporte

1
Rastreamento: Identifica cada conversa individual entre aplicações.
2
Segmentação: Divide dados em blocos aceitáveis pela rede.
3
Reunificação: Remonta os blocos na ordem original no destino.
4
Endereçamento: Usa números de porta para entregar ao processo correto.

UDP: O Protocolo "Speed"

User Datagram Protocol

O UDP é um protocolo simplificado. Ele foi desenhado para ser rápido, sacrificando a confiabilidade.

Conceito Chave: "Best Effort Delivery" (Melhor Esforço). O protocolo tenta entregar, mas se não conseguir, ele não avisa ninguém e não tenta de novo.

Por que o UDP é assim?

Sem Conexão

Não existe aperto de mão. O dado é enviado instantaneamente.

Sem ACK

Não há confirmação de recebimento. O emissor nunca sabe se o dado chegou.

Sem Ordem

Se o pacote 2 chegar antes do 1, ele é entregue assim mesmo para a aplicação.

Anatomia do Cabeçalho UDP

Apenas 8 Bytes. É o segredo da sua velocidade.

16 bitsPorta de Origem
16 bitsPorta de Destino
16 bitsComprimento Total
16 bitsChecksum (Soma de Verificação)
Checksum: O único mecanismo de erro do UDP. Se os bits chegarem fisicamente alterados, o pacote é descartado. Mas o UDP não solicita reenvio.

Quando perder dados é aceitável?

  • Streaming de Vídeo: Se um frame falhar, o vídeo continua.
  • Voz sobre IP (VoIP): É melhor um estalo no áudio do que a voz chegar 5 segundos atrasada.
  • Jogos Online: Sua posição no mapa precisa ser atualizada AGORA. Se um pacote se perdeu, a próxima atualização já trará a posição nova.

TCP: O Protocolo "Heavy Duty"

Transmission Control Protocol

O TCP é o oposto do UDP. Ele é obcecado por ordem, integridade e controle.

O que ele garante? 1. Que o dado chegou.
2. Que o dado chegou na ordem certa.
3. Que o dado não tem erros de bit.

Por que o TCP é tão robusto?

Orientado a Conexão

Estabelece uma sessão lógica antes de qualquer transmissão.

Entrega Confiável

Usa confirmações (ACKs) para garantir que cada bloco foi recebido.

Controle de Fluxo

Evita que o emissor sobrecarregue o receptor lento.

Recuperação de Erros

Reenvia automaticamente tudo o que falhou.

O Aperto de Mão: 3-Way Handshake

Nenhuma aplicação envia dados via TCP sem antes passar por este ritual:

1
SYN (Synchronize): O cliente envia um número de sequência inicial.
2
SYN-ACK: O servidor confirma o número do cliente e envia o seu.
3
ACK: O cliente confirma o número do servidor.
Agora a estrada está aberta para os dados.

Números de Sequência: A Identidade

Imagine um livro de 1000 páginas. O TCP arranca cada página e coloca um número nela antes de enviar.

Se as páginas chegarem voando e desordenadas, o destinatário usa o Sequence Number para colocar a página 1 antes da 2, a 2 antes da 3, e assim por diante.

Sem isso, seu arquivo .exe ou .pdf chegaria corrompido.

Controle de Fluxo: Janela Deslizante

O destinatário informa ao remetente quanto espaço ele tem no "estoque" (Buffer).

Receptor: "Ei, minha memória está cheia! Diminua a Janela para 100 bytes."
Emissor: "Entendido, vou mandar mais devagar."

Isso evita o descarte de pacotes por falta de memória no receptor.

Encerrando a Conversa (FIN)

O TCP também é educado para dizer adeus. Ele usa o 4-Way Handshake para fechar.

  1. Cliente: [FIN] "Não tenho mais nada pra dizer."
  2. Servidor: [ACK] "Ok, recebi seu pedido."
  3. Servidor: [FIN] "Eu também não. Tchau!"
  4. Cliente: [ACK] "Tchau! Conexão Fechada."
